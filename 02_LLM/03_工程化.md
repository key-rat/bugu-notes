### **基本概念**

1. **单轮对话：**

   - **原理：参考**[02_LLM基础](./02_LLM基础.md)
   - **提示词（Prompt）**  ：用户在对话中提交的输入内容，可以是问题、指令或任意文本。
   - **输出（Output）**  ：模型根据当前上下文生成的回复。
2. **多轮对话：**

   - **原理：** 将多次**单轮对话**记录，作为提示词的一部分，与最新的提示词一起发送给LLM，进行一次超长的**单轮对话。**

     > 从原理上看来，**多轮对话**和**单轮对话**在机制没有任何区别，都是将一次性将**提示词**发送给LLM。**多轮对话**能遵守最后一次输入的要求，是因为**上下文**的**注意力机制**。它在一定程度上可以分辨多次要求中，那些是背景信息，那些是最终要求，各个要求的优先度又是怎样的。
     >

   - **上下文（Context）：** 所有输入的提示词与LLM输出的统称。

     > 上下文内存在分层结构，使用特殊符号划分不同层次，防止内容混淆。这里的特殊符号不会在输出中显示。
     >

   - **注意力机制（Attention Mechanism）：** LLM内部**Transformer** 层的算法结构，包括点积注意力、多头注意力以及各种变体。简单理解是用于调节**上下文**内**Token**权重的算法结构。

     > **LLM缺陷，上下文相关，注意力：** LLM 对**上下文**的利用不均匀——对开头和结尾信息记忆好，对中间信息系统性遗忘。这意味着即使上下文窗口"放得下"，信息位置本身就会影响回答质量。
     >
3. **对话窗口：单轮对话**和**多轮对话**都隐含了一个概念，每次对话都发送给了同一个LLM，但短期记忆不互通。在工程化的意义上被叫做**对话窗口**，相当于编程中实例的概念。

### 关于提示词的几种用法——工程化基础

提示词是人向LLM传递意图的工具，原本从意图到产物的步骤中，多了L:LM的环节，不可避免的增加了理解与LLM处理的成本。

> 人 → LLM → 输出

- 人需要无歧义的表述背景事实，需要的成果物。

- 人需要结构化的输入，LLM需要结构化输出，以此降低期间双方的理解成本。
- LLM的能力受到输入的影响，提示词可以优化LLM输出质量。
- LLM的上下文长度有限，需要管理提示词。
- LLM只具有输出文字的能力，依托输出获取与外界交互的能力，即调用工具。
- LLM需要执行的任务可能足够复杂，需要拆解分步，因此需要流程控制；需要拆分角色，因此需要角色提示。
- LLM输出内容不稳定，需要验证，因此需要自检。

综上所述，根据提示词实际作用，大致上可以划分为以下几类：

1. 结构化表达：规范信息的组织形式，提升模型对输入的理解与输出的可解析性。

   - 结构化输入：以规范化的格式提问，明确区分背景、目标、约束等字段，减少模型对意图的猜测。

     - 注意点：规则规范本质上还是提示词，有不遵守的风险。规范过于繁杂，挤占任务空间。
     - 后续演化：规则、文档驱动、规范驱动等
   - 结构化输出：指定输出格式或 schema（如 JSON、表格、特定字段列表），便于下游程序解析或人工对比。

     - 后续演化：文档驱动、规范驱动、工具调用、UI渲染等
     - 注意点：对结构化的严格要求会降低LLM的推理质量，最好是先自由输出，再将输出内容格式化。
     - 改进：需要使用指定工具的情况，使用钩子注入提示信息。
   - 条件分支提示（输入侧）：在提示中分别说明多种情境，避免模型遇到边界情况时产生模糊输出。

     - 后续演化：TODO/任务列表、
     - 注意点：任务锚定了LLM推理方向和内容，任务或方案有问题的话，都要重来。
   - 假设显式化：要求模型在回答前列出所有前提假设，暴露隐含逻辑，方便发现偏差。

     - 注意点：幻觉问题，上下文污染
2. 上下文管理：处理模型的记忆与信息容量问题。

   - 上下文压缩：将历史对话或长文档压缩为摘要，塞入上下文，节省 token 消耗。适用于多轮长对话场景。

     - 注意点：丢失细节
     - 后续演化：滚动日志，索引，向量索引
   - 显式记忆注入：在每次请求中手动写入关键记忆（如任务列表，todo、模块契约等），弥补模型无跨会话记忆的缺陷。

     - 注意点：锚定了LLM推理方向和内容，任务或方案有问题的话，影响整体效果、甚至无法实现。
   - 滚动摘要：每轮对话结束后让模型自动总结本轮要点，用摘要替代原始历史记录滚动传入，保持信息密度。

     - 注意点：摘要有概率忽略或篡改细节。
     - 后续演化：滚动日志，索引，向量索引
3. 推理与思维引导：改变模型的思考路径，提升推理质量与深度。

   - 思维链提示（Chain-of-Thought）：要求模型逐步推导后再给出结论，而非直接输出答案。适合需要多步推理的复杂问题。

     - 注意点：基本的思维链已被内嵌入推理模型，过于详细的思维链提示会导致LLM弃用内置的思维链，导致推理效果大幅度降低。工程上的思维链倾向于任务分解而非语义分解。
     - 后续演化：TODO/任务列表
   - 树形思维（Tree-of-Thought）：让模型同时探索多条推理分支，对每条路径进行评估，再择优合并或继续深挖。

     - 注意点：分支管理困难，同一问题反复提问，得出推理分支可能不同。有过度复杂化的风险。
   - 自我一致性（Self-Consistency）：对同一问题进行多次独立推导，取一致性最高的结论。适合复杂判断或答案唯一的推理题。

     - 注意点：问法不变，可能会因为思维定势，导致结果偏差。
   - 逆向推导：从目标结果出发，反向推导所需的前提条件或中间步骤。适合规划类、诊断类问题。
   - 假设显式化：要求模型在推理前明确列出所有前提假设，暴露隐含逻辑，方便发现偏差。

     - 注意点：模型罗列的内容，可能会造成上下文污染，被默认为事实。
   - 条件分支提示（推理侧）：在提示中预设多种情境的处理路径，要求模型根据判断条件走不同分支逻辑。

     - 注意点：基本的思维链已被内嵌入推理模型，过于详细的思维链提示会导致LLM弃用内置的思维链，导致推理效果大幅度降低。工程上的思维链倾向于任务分解而非语义分解。
   - 苏格拉底式追问：要求模型主动提问澄清需求，而非直接作答，适用于需求模糊的探索性对话。

     - 注意点：模型罗列的内容，可能会造成上下文污染，被默认为事实。
   - 元认知检查：在正式回答前，先让模型确认"我是否真正理解了这个问题"，减少因误解题意导致的南辕北辙。

     - 注意点：模型罗列的内容，可能会造成上下文污染，被默认为事实。
4. 约束与质量控制：保障模型输出的可靠性、准确性与可审查性。

   - 正向约束：明确告知模型必须包含哪些要素，划定期望的输出范围。
   - 负向约束：明确告知模型不能输出哪些内容，排除干扰项。
   - 置信度要求：要求模型对每条结论标注把握程度（高/中/低，或 0–100%），区分"确定知道"与"推测"。
   - 反驳自检：要求模型在给出答案后主动寻找反例或漏洞，提升自我批评能力。
   - 长度约束：硬性限制输出的字数、段落数或条目数，避免冗余。
   - 引用来源要求：要求模型标注每个关键结论的依据或来源，而非仅输出结论，提升可信度与可审查性。
   - 示例法（Few-shot）：在提示中提供少量输入-输出示例对，引导模型学习风格、格式与推理路径。
   - 负面示例法：提供"错误示范"，告诉模型不要产出什么样的内容。与负向约束的区别：约束是规则，负面示例是样本。
   - 渐进式输出：先给出摘要或结论，再按需展开细节。适用于长文档生成或需要快速预览的场景。
   - 对比输出：要求以对比表格或并列方式呈现多个选项，便于横向比较。
   - 多方案生成：要求同时给出 N 个差异化方案，避免单点最优，为决策提供更多空间。
5. 其他

   - 改写提示词：将粗糙的提示词交给模型，要求其优化为更清晰、更有效的版本，再用优化后的版本执行任务。
   - 任务分解委托：将复杂任务拆解为若干子任务，按顺序或并行分步执行，降低单次提示的认知负荷。
   - 提示词链（Prompt Chaining）：上一步的输出作为下一步提示的输入，形成流水线，每一步专注单一子任务。
   - 多智能体分工：设计多个专职 Agent（规划者、执行者、批评者），通过角色分工协作完成复杂任务。
   - 递归提示：对输出结果反复应用同一提示进行迭代精炼，直至达到质量阈值。
   - 工具调用提示：在提示中显式声明模型可用的外部工具（搜索、代码执行、数据库查询）及其调用规范，引导模型按需使用。

**在使用提示词时，需要注意以下情况：**

- 注意点1：对结构化，对规范的严格要求会降低LLM的推理质量，最好是先输出，再将输出内容格式化或修改为规范。原因是，输出时内容模型训练资料最多，输出md文档效果最好
- 注意点2：在信息不足的情况下，模型会补足部分隐含条件，但可能和事实或需求不符，继续推理会造成逐步放大漂移。
- 注意点3：失效老旧，多个版本，或与当前任务无关的信息在上下文存在的情况，会对推理质量造成影响。
- 注意点4：基本的思维链已被内嵌入推理模型，过于详细的思维链提示会导致LLM弃用内置的思维链，导致推理效果大幅度降低。工程上的思维链倾向于任务分解，而非语义分解。
- 注意点5：过早确定方向方案，会导致模型只向确定的方向进行推理，模型不会在中途提出更好用的方案，可能会导致决策盲区。
- 注意点6：压缩上下文会造成细节损失，损失的细节会被模型用隐式猜测补足，或者LLM重新执行调查或推理流程。
- 注意点7：提示词中要求校验确认的内容，需要单独发送。模型会偷懒，要么不校验默认通过，要么将内容输出为容易通过校验的样子。

## 简单工程化

1. **工具调用：**

   - Function Call：

     - 原理：

       - 工具声明：LLM在训练中，学会将以下形式的提示词，视为可使用的工具列表。注意，各家模型规范不一，这个是最通用的形式。早期使用的是ReAct形式，

         ```
         "tools": [{
           "name": "read_file",
           "description": "读取本地文件内容",
           "input_schema": {
             "properties": { "path": { "type": "string" } }
           }
         }]
         ```
       - 工具使用：LLM在训练中，学会在需要使用工具时，输出以特殊符号包裹格式化的参数。
       - LLM在训练中学习了在需要调用工具时，输出指定格式结构， 在提示词中接收到工具的定义（名称、描述、参数类型等），推理时决定是否调用以及传什么参数。外置程序解析LLM输出内容，检测调用工具，
     - 通信规范：MCP
     - 脚本执行
2. **记忆**

   - **短期记忆**：当前**对话窗口**内，所有输入的**提示词**与LLM输出的统称，是**上下文**在持久化角度下的另一种称呼。
   - **长期记忆：** 依赖外部存储，保存**对话窗口**内的**短期记忆。**

     - 常规保存：txt、md、json、yaml等文件，postgre、neo4j等数据库。
     - **向量数据库：** 保存LLM推理过程产生的所有向量，接入LLM内部，使得LLM跳过**分词**这一步，直接开始处理**Transformer层**。

       > 模型提供商角度，**缓存：** 通过向量数据库，保留上一次LLM推理过程产生的所有向量，如果用户是多轮对话，就直接复用这部分。
